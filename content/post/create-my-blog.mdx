## 整体框架

我的毕业设计题目是一个论坛，原本打算完成后面再魔改成博客作为我自己的个人站点，当时选取的是 Nextjs 的 SSR+Koa 后端的方案，文章和评论系统选用的 quill 富文本编辑器，
后来发现博客与论坛的选型还是有些差别，对于博客这种单向输出内容的站点静态的方案似乎更有优势，也并不太需要后端提供服务，于是放弃了在毕设基础上修改的方案重新开发。

### 1. 项目框架

首先对于我的博客框架有几点要求，首先是支持 SSG，其次是纯手搓不套模版。我的主要技术栈为 React，且之前也有写过 SSR 的 Nextjs 项目，那么直接选用 Nextjs 来开发自然是最顺手的。

博客的文章展示是最核心的内容，用户评论在博客中的重要性相对较弱，因此文章发布和评论系统其实可以拆开考虑，不一定非要自己再维护一套完整后端。

#### 文章模块

博客中我不打算集成任何文章编辑器，现有的 vscode + mdx 预览插件已经满足了我的写作需求，但我需要实现文章展示。现有的文本编辑器主要是富文本编辑器和 Markdown 编辑器，
我的博客主要内容应该是技术写作，会有较多的代码展示且可能涉及跨平台分享，所以文章内容方案会选用 markdown 实现。

#### 为什么选 MDX

进一步了解后发现 MDX 可以混合编写 Markdown 和 JSX，并且可以在 Markdown 中导入组件，这意味着我可以在文章中插入自定义图表、代码块等功能，能让文章更加生动有趣。
在寻找内容管理工具时发现了 ContentLayer 这款工具，非常契合我的使用场景，集成完成后发现因为赞助的问题该项目已经暂停维护，但是暂时还没发现有其他方案比 ContentLayer 更加方便，
于是暂时转到了 ContentLayer 的社区管理分支 [ContentLayer2](https://github.com/timlrx/contentlayer2)，如果后续出现了更合适的方案再考虑迁移。

#### Contentlayer 里配置了什么

具体接入时其实也没有想象中那么复杂，核心就是两件事。第一件事是告诉 Contentlayer 我的文章长什么样，也就是在 `contentlayer.config.js` 里定义 frontmatter 的字段结构。

比如标题、描述、日期、标签、头图、作者这些信息都提前声明好，这样写文章时如果漏了关键字段，编译阶段就能直接报错，不至于等到页面渲染时才发现少东西。

除了这些基础字段外，我还额外定义了两个计算属性，一个是 `slug`，另一个是 `slugAsParams`。这样后面在文章列表页、详情页动态路由和标签筛选这些地方，
都可以直接使用 Contentlayer 编译出来的结果，不需要自己再反复处理文件路径。

```js
const computedFields = {
	slug: {
		type: 'string',
		resolve: (doc) => `/${doc._raw.flattenedPath}`,
	},
	slugAsParams: {
		type: 'string',
		resolve: (doc) => doc._raw.flattenedPath.split('/').slice(1).join('/'),
	},
};

export const Post = defineDocumentType(() => ({
	name: 'Post',
	filePathPattern: `**/*.mdx`,
	contentType: 'mdx',
	fields: {
		title: { type: 'string', required: true },
		description: { type: 'string' },
		date: { type: 'date', required: true },
		tags: { type: 'list', of: { type: 'string' } },
		image: { type: 'string', required: false },
		authors: {
			type: 'list',
			of: { type: 'string' },
			required: true,
			default: ['haiminovo'],
		},
	},
	computedFields,
}));
```

#### MDX 渲染链路

第二件事则是让 MDX 的展示体验更适合技术博客。这里我挂了 `remark-gfm` 来支持表格、任务列表这类 GitHub 风格语法，又配了 `rehype-slug`、`rehype-autolink-headings`
和 `rehype-pretty-code` 来分别处理标题锚点和代码高亮。这样文章里的标题天然就带 `id`，后面目录导航组件也能直接复用这份结果，算是顺手把两件事一起解决了。

真正渲染文章时，Next 页面层负责从 `allPosts` 中拿到对应文章数据，再把 `page.body.code` 交给自定义的 `Mdx` 组件。`Mdx` 内部通过 `useMDXComponent`
把编译后的代码转成 React 组件，同时把自己写的 `Callout`、`Card`、代码复制按钮之类的组件映射进去。这样写作时还是在写 MDX，但展示层已经可以完全按博客自己的样式体系来控制。

#### 评论模块

因为是静态的页面，并且我不想依靠后端来构建评论系统，虽然带后端的评论系统可以实现邮件推送等功能但是感觉有些臃肿，因此一切需要额外部署服务的评论系统都先排除。看了一下基于 github 的评论实现还挺多的，最后因为文章 action 的原因选择了 Giscus，
并且其他的评论系统也似乎很久没有更新了，Giscus 上还能看到近几个月的更改。

#### 评论区和主题同步

接入 Giscus 后还有一个小问题需要处理，就是它本身也有亮色和暗色主题，而我的博客又支持手动切换主题。如果不额外同步，页面切黑了评论区还是亮的，看起来就会非常割裂。

我的处理方式是把当前主题存进 `localStorage`，切换主题时除了修改 `html` 上的 `dark` 类名外，再派发一个自定义事件，让评论组件也跟着更新自己的 `theme` 属性。

这样做的好处是结构比较松耦合，暗黑模式按钮不需要直接操作 Giscus 实例，评论组件也只负责监听主题变化并重新渲染，逻辑上还算清楚。顺手一提，我前面那篇关于监听
`localStorage` 的文章，其实就是在折腾这块时顺便记下来的。

### 2. 部署方案

大二时薅羊毛一百不到买了十年的轻量应用服务器，放着不用就太浪费了，用来挂个博客绰绰有余，并且做这个博客也是有上手实操一下 SEO 的想法在，因此就不挂到 Vercel 或者 Github Pages 了。

#### 为什么走静态导出

既然最终是部署到自己的服务器上，那么在 Next 这边我就尽量往纯静态导出的方向靠。现在项目配置里用的是 `output: 'export'`，这样构建完成后可以直接产出静态文件，
对于博客这种以读为主、更新频率也不高的站点来说已经足够用了。为了避免动态路由在静态部署环境中刷新 404，我还顺手加上了 `trailingSlash: true`。

```js
import { withContentlayer } from 'next-contentlayer2';

const nextConfig = {
	output: 'export',
	trailingSlash: true,
	images: {
		loader: 'akamai',
		path: '',
		unoptimized: true,
	},
};

export default withContentlayer(nextConfig);
```

图片这里也额外处理了一下。因为是静态导出，Next 自带的图片优化服务并不能像部署在官方运行时那样直接使用，所以我把图片配置切成了 `unoptimized`，
并配合静态导出环境可用的 loader 方案，优先保证生成后的站点能稳定跑起来。等后面如果有精力，再研究一套更细一点的图片资源策略。

#### 自动化部署

自动化部署这一块其实是我一开始就很想补上的内容。大四时两家实习过的公司都有自动化的 CI/CD 流程，写完代码 push 上去等它自己打包发布，体验确实比手动登录服务器舒服太多。

不过这类能力自己从零搭起来时才发现，真正麻烦的不是写几行脚本，而是要把构建、上传、覆盖、回滚这些环节都串顺。

所以当前这个博客项目的思路是先把“本地写作 -> Contentlayer 编译 -> Next 静态导出 -> 服务器托管”这条主链路打通，至少先保证站点能稳定更新。自动化发布后面肯定还会做，
只是现阶段我更希望先把页面、文章系统和整体体验打磨完整，再去补 CI/CD 这块工程化细节。

### 3. 布局及样式

#### 响应式布局方案

对于博客界面，响应式是肯定要做的，界面整体打算做三栏响应式布局，pc 上展示为三栏，平板上展示为两栏，移动端展示单栏。侧边栏定宽，内容栏自适应，使用媒体查询和 flex 布局实现界面适配。

#### 页面整体布局

现在项目里的布局实现基本也是沿着这个方向在做。最外层是一个统一的 `RootLayout`，顶部放导航栏，中间是左右侧边栏加主内容区，右侧挂标签、目录和返回顶部，
左侧放个人信息和站点信息。到了较小屏幕下再通过断点逐步隐藏部分侧栏内容，把主要空间让给正文和文章列表，整体结构会比一开始就只做单栏更灵活一点。

#### 首页和文章页的区分

首页和文章页在布局上也做了区分：首页偏展示，用横幅、头像和近期更新做一个轻量的落地页；文章详情页则更强调阅读体验，所以目录、评论区和返回顶部这些辅助组件都会围着正文来服务。

#### CSS 方案

毕业前的最后一段实习接触到了 tailwindCSS 这种原子化 CSS，不需要编写样式名，不用书写冗长的 BEM 规范，不用关心一层套一层的样式优先级覆盖，简直爽到嗨。这里不必多说，
直接选用 tailwindCSS，没有体验过的强烈推荐尝试一下原子化 CSS。

##### 默认主题

很早之前就想实现一个可以任意调整主题颜色的功能，这次实现的思路是根据传入的基础色使用 js 生成由基础色到纯白以及由基础色到纯黑的 1 到 10 十个级别的共 20 种颜色给到 tailwindCSS，
在需要使用主题色的元素中尽量只使用这 20 种颜色，这样再更换主题色时只需要调整输入的基础色就可以完成整套颜色样式的更新。现在这部分逻辑我直接写在 `tailwind.config.mjs` 里，
这样启动和构建时就能直接生成主题色。

```js
const hexToRgb = (str) => {
	let hexs = null;
	const reg = /^\#?[0-9A-Fa-f]{6}$/;
	if (!reg.test(str)) {
		throw new Error('Invalid color value in tailwind.config.mjs');
	}
	str = str.replace('#', '');
	hexs = str.match(/../g);
	return hexs?.map((item) => parseInt(item, 16));
};

const generateColor = (str, mode = 'light') => {
	let basicColor = hexToRgb(str);
	if (!basicColor) return [];
	if (mode === 'dark') basicColor = basicColor.map((item) => item - 100);
	const stepArr = basicColor.map((item) => (mode === 'dark' ? (item / 10).toFixed(0) : ((255 - item) / 10).toFixed(0)));
	return new Array(10).fill(undefined).map((_, index) => {
		const targetColor = basicColor.map((item, colorIndex) => {
			return mode === 'dark'
				? item - index * +stepArr[colorIndex] < 0
					? 0
					: item - index * +stepArr[colorIndex]
				: item + index * +stepArr[colorIndex] > 255
					? 255
					: item + index * +stepArr[colorIndex];
		});
		return `rgb(${targetColor[0]},${targetColor[1]},${targetColor[2]})`;
	});
};
```

#### 颜色导入方式

在 `tailwind.config.mjs` 中将自定义颜色导入：

```js
const basicColor = '#e2e5e8';
const themeColors = {};

generateColor(basicColor).forEach((item, index) => {
	themeColors[`custom-color-${index + 1}`] = item;
});

generateColor(basicColor, 'dark').forEach((item, index) => {
	themeColors[`custom-color-dark-${index + 1}`] = item;
});

const config = {
	content: [
		'./src/pages/**/*.{js,ts,jsx,tsx,mdx}',
		'./src/components/**/*.{js,ts,jsx,tsx,mdx}',
		'./src/app/**/*.{js,ts,jsx,tsx,mdx}',
	],
	theme: {
		extend: {
			colors: {
				'font-light': '#5a6a7a',
				'font-normal': '#3a4a5a',
				'font-strong': '#1a2a3a',
				'font-light-dark': 'rgba(255,255,255,0.9)',
				'font-normal-dark': 'rgba(255,255,255,0.7)',
				'font-strong-dark': 'rgba(255,255,255,0.5)',
				...themeColors,
			},
		},
	},
};
```

按这个方式生成的样式对所选基础色的要求比较高，需要在适当的区间中选择，否则会导致深色模式中一团黑，或者浅色模式亮得晃眼睛，后续会考虑重写一下颜色生成的方案并把调色盘放出来。

##### 暗黑主题

tailwindCSS 有自带的 [darkmode 方案](https://tailwindcss.com/docs/dark-mode)，这里整体上也是沿用了同样的思路，只不过主题切换的控制权放在我自己的按钮组件里。

初始化时先从 `localStorage` 读取用户上次选择的主题，再给 `html` 节点增删 `dark` 类名，后续点击切换时同步更新本地缓存和页面状态。

对于暗黑模式的字体，我没有像默认主题那样手动再配一套复杂的色板，而是更多使用纯白加透明度来区分层级。这样做虽然朴素，但是和当前这种偏浅灰、低饱和的整体主题搭起来观感还不错，
并且维护起来也轻松一些。

### 4. 文章展示细节

#### 数据是怎么流转起来的

博客真正写起来之后我才发现，文章系统这块除了“能展示”之外，还有很多零碎但影响体验的细节。比如文章列表页其实直接消费的是 `allPosts`，标签页也是在同一份数据上做过滤，
侧边栏里的标签数量统计同样来自 Contentlayer 编译后的结果。也就是说只要新增一篇带标签的文章，列表、标签筛选、站点信息这些模块都会自动更新，不需要我再去维护额外的数据文件。

#### 详情页展示结构

文章详情页则是另一层流程。动态路由会先根据 `slugAsParams` 找到对应文章，再生成页面的 `title`、`description` 和 `keywords`，这样至少在基础的 SEO 信息上是完整的。
正文上方的头图、文章简介、正文内容和评论区也都按这个流程拼起来，结构上比较清楚，后续再继续扩展阅读量、上一篇下一篇之类的功能也会比较方便。

#### 这些小功能为什么值得做

另外像标题锚点、代码复制、代码高亮、目录导航这些看上去不大的功能，其实都属于“单独拿出来没什么，但少了就会觉得差点意思”的部分。做这个博客到现在最大的感受之一就是，
博客这种东西从来都不是把文章渲染出来就结束了，真正决定使用体验的往往就是这些边边角角的小组件和小交互。

### 5. 小结

目前这个博客项目还远远算不上做完，像自动化部署、更完善的主题系统、图片资源处理甚至更顺手的写作体验，后面都还有很多可以继续折腾的空间。

不过至少到现在为止，整体的技术路线已经比较稳定了：Next.js 负责页面和路由，Contentlayer2 负责把 MDX 文章编译成可消费的数据，Giscus 负责评论，TailwindCSS 负责样式体系，
再配上一套自己手搓的小组件，基本就能把一个个人博客需要的核心能力搭起来。

对我来说这次最大的收获倒不只是“又做了一个站”，而是终于把“想要一个真正属于自己的内容输出阵地”这件事往前推了一大步。后面如果又踩到什么坑，或者把哪些细节补得更顺手了，
大概率还会继续写文章记录下来。
